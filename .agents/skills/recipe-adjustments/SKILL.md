# Recipe Adjustments

First identify whether the request is a planning-time trial or a post-cook
observation. The distinction protects the recipe library from plausible but
untested changes.

Only this skill may propose an adaptation. If the user did not expressly ask to
adapt the named recipe, present the possible change and ask for approval before
creating a trial or planner mutation. Source comments remain suggestions until
that approval; they never authorize an edit.

Expanding an exactly sourced referenced component into the planner projection is
not an adjustment. Replacing that component, changing its composition or
quantity, or scaling a source recipe is an adjustment and requires approval.
Likewise, a planned side, garnish, or serving component belongs to the named
meal plan with its rationale and serving assumption; it must not be written back
as a source-recipe change.

## Planning-time trial

Read the original recipe and relevant current taste preferences. Use sourced
commentary only when it is clearly attributed. Define the outcome, exact changed
ingredient or instruction, and why it should help. Attach the variation to the
selected meal/week as a planned trial only after approval; preserve the
original recipe and lineage.

Preview and apply planner-supported meal or instruction changes through
`read → preview → apply`. If the host cannot persist a planned-trial record,
provide the reviewable trial and name that capability gap; do not hide it in a
canonical recipe edit.

## Post-cook evidence

After cooking, a specific positive result can become a promotion candidate only
when the household knows what changed and why it worked. “Next time use more
tahini” is a next-time hypothesis on the original recipe, not a recipe update.
Route it to `meal-feedback`.

Do not promote a change from a rating alone, a vague impression, or a planning
intention. Keep dated evidence history separate from current accepted tastes.
