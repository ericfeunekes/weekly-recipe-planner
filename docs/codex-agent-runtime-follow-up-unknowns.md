## Closed Dispositions

### 1. Several dependent calls on the installed app-server

**Answer.** Yes. On the tested 0.142.5 app-server, one ephemeral planner turn invoked `planner.create_recipe` as call A, received host result `{recipeId:"recipe-123"}`, then invoked `planner.schedule_recipe` as call B with that exact ID and completed normally. The app-server made three Responses requests: initial, after A's tool output, and after B's tool output. This proves transport sequencing and output propagation; the scripted mock model does not claim autonomous reasoning quality.

**Evidence.** `/tmp/codex-agent-capability-spike/captures/result.json` contains both `item/tool/call` requests, host results, and terminal `turn/completed`. `/tmp/codex-agent-capability-spike/captures/responses-requests.jsonl` records the three planner model requests. Official docs and current source independently define the [dynamic-tool lifecycle](https://learn.chatgpt.com/docs/app-server#dynamic-tool-calls-experimental), while the disposable spike exercised it on the exact installed binary.

**Closes because.** The load-bearing several-dependent-call primitive is observed on a concrete current artifact. The update-aware gate requires every later Codex build to preserve it before agent readiness. Real authenticated model behavior remains an opt-in acceptance test, not a planning unknown.

### 5. Dynamic-tool wire and host fencing

**Answer.** Initialize once per connection with `capabilities.experimentalApi = true`, then send `initialized`. `thread/start` carries `ephemeral: true`, the fixed cwd, `environments: []`, per-thread config/permissions, and function or namespace `dynamicTools`. On `item/tool/call`, the host validates `{threadId, turnId, callId, namespace?, tool, arguments}` and replies to the separate JSON-RPC request `id` with `{contentItems:[{type:"inputText",text}],success}`. `item/completed`, not intent or `item/started`, is authoritative. The host binds the first `(threadId, turnId, callId)` to tool name, canonical argument hash, durable household turn, and completion token; exact replay receives the recorded outcome, changed-payload or reused identity fails, and unknown tools fail closed. A local call gets a 30-second host deadline. On deadline or connection loss the host records failure, returns `success:false` when the channel is usable, interrupts the turn, rotates the completion token, and ignores any late response. App-server's `turn/interrupt` is turn-level; there is no dynamic-call cancellation method or built-in duplicate/idempotency guarantee.

**Evidence.** The official [app-server protocol](https://learn.chatgpt.com/docs/app-server#protocol), [initialization contract](https://learn.chatgpt.com/docs/app-server#initialization), and [dynamic-tool lifecycle](https://learn.chatgpt.com/docs/app-server#dynamic-tool-calls-experimental) match the installed experimental JSON schema generated from Codex 0.142.5. Tested-version source shows dynamic responses wait without a timer and that duplicate `callId` values can overwrite pending senders: [dynamic response handling](https://github.com/openai/codex/blob/26de83050b20f7e0ee211b9739e52ae00ce8032a/codex-rs/app-server/src/dynamic_tools.rs#L16-L55) and [duplicate-call handling](https://github.com/openai/codex/blob/26de83050b20f7e0ee211b9739e52ae00ce8032a/codex-rs/core/src/tools/handlers/dynamic.rs#L172-L209).

**Closes because.** The adapter has one exact request/response identity model and one fail-closed host policy for timeout, cancellation, duplicates, changed payloads, connection loss, and late responses. The 30-second local deadline is an implementation constant, not a vendor behavior assumption.

### 9. Ephemeral persistence and retention

**Answer.** `ephemeral: true` makes the thread in-memory and pathless, skips the local thread record/rollout, and excludes it from persisted search/read/list. It remains live until unload/process exit. It does **not** make `CODEX_HOME` fileless: the installed app-server creates state, log, goal, and memory SQLite scaffolding, and its trace-level `logs_2.sqlite` captured the disposable tool arguments/results even though the `threads`, `thread_dynamic_tools`, and `agent_jobs` tables stayed empty. Tested 0.142.5 continuously caps each thread/process partition at 1,000 rows and roughly 10 MiB; startup maintenance deletes rows older than ten days, so a continuously running process can retain older rows between restarts. The dedicated home is therefore a private runtime-content store, not merely an auth directory. Standard non-Azure Responses requests use `store: false`, but that still does not promise zero service-side operational retention, prompt caching, or hosted-search retention. Production keeps the dedicated home `0700`, does not copy its logs into application logs/backups, documents the active build's observed retention behavior, and treats provider/workspace retention as a separate deployment disclosure. The compatibility gate re-inventories runtime files and content-retention behavior whenever Codex changes.

**Evidence.** [Ephemeral session construction](https://github.com/openai/codex/blob/54c44b9ed4c7d6d1ec9bf7897bb76f6411d8e033/codex-rs/core/src/session/session.rs#L579-L668) skips thread persistence, while [Responses request construction](https://github.com/openai/codex/blob/54c44b9ed4c7d6d1ec9bf7897bb76f6411d8e033/codex-rs/core/src/client.rs#L898-L915) distinguishes standard and Azure `store` behavior. The installed-version probe found zero persisted thread/dynamic-tool/job rows but 256 runtime log rows, including eight containing its synthetic call content. Tag `rust-v0.142.5` fixes local log retention in `codex-rs/state/src/runtime.rs:97-98` and `runtime/logs.rs:3`.

**Closes because.** The architecture no longer equates ephemeral execution with universal no-retention. Local cleanup/proof and provider disclosure are fixed, and no implementation branch depends on an unavailable stronger promise.

### 6. Planner tool and mutation contract

**Answer.** One deployment-owned command registry generates the HTTP validator, exact shipped `PlannerCommand` schema, and default-deny `AuthorityManifest`. The manifest fixes a schema/hash version, command scope and exposure (`ordinary` or `explicit_foreground`), permanently denied operation kinds, and limits: 32 tool calls per turn, 16 commands per batch, 64 KiB arguments, 128 KiB results, and 20 recent events. Register one `planner` namespace with exactly six functions: `authority({})`, bounded `read(query)`, pure `preview({basePlannerVersion, operations[1..16]})`, `apply({basePlannerVersion, operation, readback})`, `apply_batch({basePlannerVersion, operations[1..16], readback})`, and `refresh({conflictedBasePlannerVersion, query})`. Read queries are limited to workspace, week, meal, and a sanitized history tail; they never return undo snapshots. The host, not the model, supplies actor, provenance, turn/token/call identity, and the stable request ID.

`PlannerOperation` is exactly `{weekId: WeekId|null, command: PlannerCommand}`, with null permitted only for registry-declared workspace commands. `ReadQuery` is exactly one of `{kind:"workspace"}`, `{kind:"week",weekId}`, `{kind:"meal",weekId,mealId}`, or `{kind:"history",afterSequence?,limit:1..20}`. Preview creates no receipt/event/version/staging token and omits generated IDs until apply readback.

All results use a versioned bounded envelope containing `ok`, host call ID, current planner/sync versions, server time, and typed data or a sanitized error with at most 20 field errors and one of `revise_new_call|refresh_new_call|wait_same_call|new_foreground_turn|none`. Error codes are `INVALID_ARGUMENTS|NOT_AUTHORIZED|VERSION_CONFLICT|DOMAIN_REJECTED|DUPLICATE_MISMATCH|CALL_IN_PROGRESS|CALL_CANCELLED|CALL_TIMED_OUT|LATE_CALL|TURN_NOT_RUNNING|INTERNAL_ERROR`. Follow-up landing generalizes the family kernel to `applyPlannerOperations(tx, {operationKind, requestId, basePlannerVersion, actor, provenance, operations})`; existing HTTP/chat wrappers always pass one operation, while batch passes 1–16. The whole list reduces against one in-memory candidate and commits one receipt, event/change set, undo basis, and version increment or nothing.

**Evidence.** The current plan fixes service ownership and the one-command caller-owned transaction at `scratch/family-readiness-remediation-plan.md:91-104` and `:156-191`, and explicitly defers multi-command input at `:167`. The existing `site/lib/planner-command-contract.ts` supplies the strict schema/validator pattern but is evidence rather than the future vocabulary. The follow-up's multiple-effect and batch semantics are already fixed at `scratch/codex-agent-runtime-follow-up-phase.md:81-135`.

**Closes because.** Tool names, arguments, limits, results, command authority, and dependency direction now have one implementation target. The current family release remains single-command; only the follow-up adds the generalized array core and batch wrapper.

### 7. Durable call/effect lifecycle

**Answer.** Extend `chat_turns` with `mode` (`normal|recovery`), `completion_token_hash`, `accepted_effect_count`, `last_effect_sequence`, `recovery_of_turn_id`, and terminal outcome `completed_no_effect|completed_with_effects|failed_no_effect|failed_after_effect|interrupted_no_effect|interrupted_after_effect|recovery_completed|recovery_failed`. Add one `planner_tool_calls` ledger keyed by `(turn_id, tool_call_id)` with sequence, token hash, tool/argument hash, status (`running|succeeded|rejected|cancelled|timed_out|abandoned`), result/effect codes, operation/request/event identities, base/result planner versions, an immutable bounded sanitized replay envelope, and timestamps. Reserve only against a running turn/current token. Exact duplicates replay; changed name/arguments fail. An accepted apply/batch commits workspace, receipt, event, tool row, effect sequence, and turn effect count together before responding. Separate accepted calls are separate effects; an atomic batch is one effect.

Startup marks pre-commit running calls abandoned and the turn interrupted; post-commit calls replay their succeeded ledger row. Reply failure after effects persists `failed_after_effect` and a canonical visible failure. Retry with zero effects creates a new normal turn; Retry after any effect creates a recovery turn with no web or planner tools and only reconstructs the reply from durable effects/current readback. Late/revoked-token calls and terminal replies cannot mutate state.

**Evidence.** The current bounded lifecycle and restart fencing are at `scratch/family-readiness-remediation-plan.md:193-221`. The follow-up fixes separately committed dependent effects and recovery-only Retry at `scratch/codex-agent-runtime-follow-up-phase.md:92-126`; its proof matrix requires process exits around receipt, commit, response, and terminal reply at `:183-185`.

**Closes because.** Every call identity, accepted effect, crash point, terminal outcome, startup action, and Retry mode maps to one durable row/transition and one transaction boundary.

### 8. Research candidate boundary

**Answer.** Research emits strict `additionalProperties:false` JSON no larger than 32 KiB: `{schemaVersion:1, candidateId, source:{kind:"web",identity,url,retrievedAt}, title, yieldText?, steps[1..32]}`. Source identity/title are 1–200 characters, URL is HTTP(S) and at most 2,048 characters, yield is at most 80, and candidate ID/retrieval time are host materialized. Each step has `inputs[0..12]` with amount 1–80 and ingredient 1–160, instruction 1–1,000, and optional duration 1–86,400 seconds. There are no HTML, Markdown, page-body, excerpt, arbitrary metadata, or attachment fields. The broader provenance enum may later include app-owned non-web kinds, but embedded research always emits `web`.

The planner context sees only this JSON. No candidate/library authority table is added. An accepted typed planner command materializes authoritative IDs and stores provenance inside the week/meal-local `RecipeSnapshot`; before acceptance the shared transcript keeps only candidate ID, title, compact source identity/URL, and step count. Application logs do not retain the full candidate.

**Evidence.** The context split and week/meal-local snapshot rule are fixed at `scratch/codex-agent-runtime-follow-up-phase.md:69-91`. Existing strict size/exact-key validation lives at `site/bridge/validation.mjs:9-25` and `:84-156`; meal-local, non-library recipe semantics are grounded in `scratch/functional-spine.md:41-59` and `:99-107`.

### 10. Global-client transport

**Answer.** Use HTTP/1.1 over the fixed socket `$HOME/meal-planner/run/global-codex.sock`; private loopback is not an alternative in this phase. The deployment-owned run directory is current-UID `0700` and the socket is `0600`. Same-UID access is the accepted trust boundary; `globalCodex` is ingress provenance, not cryptographic process identity. Bind only after store/service readiness. On `EADDRINUSE`, never delete a non-socket, symlink, wrong-owner, or wrong-directory-mode path. Graceful shutdown closes and verifies removal. UDS failure disables only global ingress, and the client never falls back to TCP or SQLite.

**Evidence.** Node 22.13 supports [`http.request({socketPath})`](https://nodejs.org/download/release/v22.13.0/docs/api/http.html#httprequestoptions-callback), filesystem IPC paths, `lstat`, `Stats.isSocket()`, and `chmod`. **Closes because.** Listener, path, permissions, admission, cleanup, stale-socket behavior, availability, and the same-UID residual risk have one unconditional answer.

### 11. Thin-client dependency and ingress contract

It is never mounted in `/api`, Vite/Vinext/front-controller proxying, or the old bridge; it rejects Origin/fetch-metadata/upgrade/proxy behavior and emits no CORS. The batch request is strict JSON (160 KiB maximum): `{contractVersion:1, requestId, basePlannerVersion, commands[1..16]}` using the exact shared `PlannerOperation` schema. Actor/principal/provenance, state, transcript, prompts, DB/SQL/socket/upstream fields are forbidden. The adapter injects `{actorClass:"codex",actorSource:"global",admission:"same_uid_uds_v1"}` and calls `PlannerMutationService.applyBatch`; it never calls the store directly.

Receipt identity is `("global_codex_apply_planner_batch_v1", requestId)`, hashing contract/operation kind, base version, and ordered canonical commands. Same ID/hash replays the immutable decision with fresh readback; changed payload fails; schema-invalid input reserves nothing; accepted/conflict/domain-rejected decisions reserve the ID. Transport uncertainty retries the same ID, while a deliberate post-refresh attempt uses a new ID. The CLI imports only shared DTO/schema code and `node:http`; it has no server composition, `node:sqlite`, database path, SQL, browser route, or actor-setting dependency.

**Evidence.** Family OCC, receipts, and authoritative readback are fixed at `scratch/family-readiness-remediation-plan.md:154-191`; the follow-up batch semantics are at `scratch/codex-agent-runtime-follow-up-phase.md:127-136`. Current `site/bridge/server.mjs` and `validation.mjs` demonstrate the browser transport/schema seam being replaced, not a reusable authority.
