### 5. Dynamic-tool wire and host fencing

**Historical answer.** Initialize once per connection with `capabilities.experimentalApi = true`, then send `initialized`. `thread/start` carries `ephemeral: true`, the fixed cwd, `environments: []`, per-thread config/permissions, and function or namespace `dynamicTools`. On `item/tool/call`, the host validates `{threadId, turnId, callId, namespace?, tool, arguments}` and replies to the separate JSON-RPC request `id` with `{contentItems:[{type:"inputText",text}],success}`. `item/completed`, not intent or `item/started`, is authoritative. The host binds the first `(threadId, turnId, callId)` to tool name, canonical argument hash, durable household turn, and completion token; exact replay receives the recorded outcome, changed-payload or reused identity fails, and unknown tools fail closed. A local call gets a 30-second host deadline. On deadline or connection loss the host records failure, returns `success:false` when the channel is usable, interrupts the turn, rotates the completion token, and ignores any late response. App-server's `turn/interrupt` is turn-level; there is no dynamic-call cancellation method or built-in duplicate/idempotency guarantee.

**Evidence.** The official [app-server protocol](https://learn.chatgpt.com/docs/app-server#protocol), [initialization contract](https://learn.chatgpt.com/docs/app-server#initialization), and [dynamic-tool lifecycle](https://learn.chatgpt.com/docs/app-server#dynamic-tool-calls-experimental) match the installed experimental JSON schema generated from Codex 0.142.5. Tested-version source shows dynamic responses wait without a timer and that duplicate `callId` values can overwrite pending senders: [dynamic response handling](https://github.com/openai/codex/blob/26de83050b20f7e0ee211b9739e52ae00ce8032a/codex-rs/app-server/src/dynamic_tools.rs#L16-L55) and [duplicate-call handling](https://github.com/openai/codex/blob/26de83050b20f7e0ee211b9739e52ae00ce8032a/codex-rs/core/src/tools/handlers/dynamic.rs#L172-L209).

**Historical closure.** The adapter has one exact request/response identity model and one fail-closed host policy for timeout, cancellation, duplicates, changed payloads, connection loss, and late responses. The 30-second local deadline is an implementation constant, not a vendor behavior assumption.

### 9. Ephemeral persistence and retention

**Historical answer.** `ephemeral: true` makes the thread in-memory and pathless, skips the local thread record/rollout, and excludes it from persisted search/read/list. It remains live until unload/process exit. It does **not** make `CODEX_HOME` fileless: the installed app-server creates state, log, goal, and memory SQLite scaffolding, and its trace-level `logs_2.sqlite` captured the disposable tool arguments/results even though the `threads`, `thread_dynamic_tools`, and `agent_jobs` tables stayed empty. Tested 0.142.5 continuously caps each thread/process partition at 1,000 rows and roughly 10 MiB; startup maintenance deletes rows older than ten days, so a continuously running process can retain older rows between restarts. The dedicated home is therefore a private runtime-content store, not merely an auth directory. Standard non-Azure Responses requests use `store: false`, but that still does not promise zero service-side operational retention, prompt caching, or hosted-search retention. Production keeps the dedicated home `0700`, does not copy its logs into application logs/backups, documents the active build's observed retention behavior, and treats provider/workspace retention as a separate deployment disclosure. The compatibility gate re-inventories runtime files and content-retention behavior whenever Codex changes.

**Evidence.** [Ephemeral session construction](https://github.com/openai/codex/blob/54c44b9ed4c7d6d1ec9bf7897bb76f6411d8e033/codex-rs/core/src/session/session.rs#L579-L668) skips thread persistence, while [Responses request construction](https://github.com/openai/codex/blob/54c44b9ed4c7d6d1ec9bf7897bb76f6411d8e033/codex-rs/core/src/client.rs#L898-L915) distinguishes standard and Azure `store` behavior. The installed-version probe found zero persisted thread/dynamic-tool/job rows but 256 runtime log rows, including eight containing its synthetic call content. Tag `rust-v0.142.5` fixes local log retention in `codex-rs/state/src/runtime.rs:97-98` and `runtime/logs.rs:3`.

**Historical closure.** The architecture no longer equates ephemeral execution with universal no-retention. Local cleanup/proof and provider disclosure are fixed, and no implementation branch depends on an unavailable stronger promise.

### 6. Planner tool and mutation contract

**Historical answer.** One deployment-owned command registry generates the canonical HTTP/runtime validator, a compact model-facing `PlannerCommand` guidance projection, the `preview`/`apply` command-type union, the current bounded-chat provider-strict projection/normalizer, and an internal default-deny `AuthorityManifest`. Canonical optional fields remain optional and non-null; the legacy provider projection alone renders those fields required-but-nullable for strict structured output and then removes null sentinels before canonical validation. The manifest is not model-facing. To remain below Codex 0.142.5's dynamic-tool compaction boundary, the shipped planner schema shows every allowed command discriminator and the generated tool description lists each alternative's required top-level fields; nested value shapes, enums, limits, optionality, and extra-field rejection remain canonical host-validator authority. An invalid model call is rejected without planner, event, or version effects. The manifest fixes a schema/hash version, command scope and exposure (`ordinary` or `explicit_foreground`), permanently denied operation kinds, and limits: 32 tool calls per turn, 16 commands per apply, 64 KiB arguments, 128 KiB results, and 20 recent events.

Register one `planner` namespace with exactly three functions: bounded `read({query})`, pure `preview({basePlannerVersion, operations[1..16]})`, and transactional `apply({basePlannerVersion, operations[1..16], readback})`. `apply.readback` is exactly one `ReadQuery` and selects the bounded canonical post-acceptance/replay projection returned with the decision; a conflict returns current versions plus `refresh_new_call`, and the model uses `planner.read` for the authoritative refresh. One operation is the single-command case; 2–16 are reduced and committed as one atomic batch. `read` handles both ordinary reads and authoritative refresh after `VERSION_CONFLICT`; no separate refresh tool exists. Read queries are limited to workspace, week, meal, and a sanitized history tail, and never return undo snapshots, transcript, chat turns, receipts, or before-state data. The host, not the model, supplies actor, provenance, turn/token/call identity, and the stable request ID.

`PlannerOperation` is exactly `{command: PlannerCommand}`. The exact shipped command union carries its own week or workspace identity, so there is no duplicate outer `weekId` and no nullable scope convention. Latest-only undo remains the existing guarded recovery service and is not a `PlannerCommand`, preview/apply operation, or model-visible agent action; an accepted batch is one event that the existing UI/service can later undo as a unit. `ReadQuery` is exactly one of `{kind:"workspace"}`, `{kind:"week",weekId}`, `{kind:"meal",weekId,mealId}`, or `{kind:"history",afterSequence?,limit:1..20}`. Preview is pure: it validates the caller's current base version, reduces against an in-memory candidate, and returns either `previewed` with ordered per-operation `{operationIndex,summary,target,changes}` entries, `version_conflict`, or `domain_rejected` with the zero-based failing `operationIndex`. It creates no receipt, event, version, staging token, or durable ID and exposes no generated IDs or candidate state. Before returning preview outcomes, the host replaces every occurrence of an ID created by the throwaway reducer in `summary`, `target`, or `changes` with the exact text `[generated after apply]`. A later operation that needs a generated ID must be a separate apply after authoritative readback.

All results use a versioned bounded envelope containing `ok`, host call ID, current planner/sync versions, server time, and typed data or a sanitized error with at most 20 field errors and one of `revise_new_call|refresh_new_call|wait_same_call|new_foreground_turn|none`. Error codes are `INVALID_ARGUMENTS|NOT_AUTHORIZED|VERSION_CONFLICT|DOMAIN_REJECTED|DUPLICATE_MISMATCH|CALL_IN_PROGRESS|CALL_CANCELLED|CALL_TIMED_OUT|LATE_CALL|TURN_NOT_RUNNING|INTERNAL_ERROR`. Follow-up landing generalizes the family kernel to `applyPlannerOperations(tx, {operationKind, requestId, basePlannerVersion, provenance, operations})`; existing HTTP/chat wrappers always pass one operation, while batch passes 1–16. The host-generated provenance union is exactly `{actorClass:"household",actorSource:"browser",admission:"same_origin_http_v1"}`, `{actorClass:"codex",actorSource:"embedded_legacy",admission:"structured_output_v1"}`, `{actorClass:"codex",actorSource:"embedded",admission:"app_server_dynamic_v1"}`, or `{actorClass:"codex",actorSource:"global",admission:"same_uid_uds_v1"}`. The display actor remains `Household|Codex`, derived from `actorClass`; a forward migration stores provenance for every new event and maps existing Household/Codex rows to the browser/embedded-legacy forms.

The whole list reduces against one in-memory candidate. A one-operation acceptance preserves the existing event command, summary, target, and changes. A 2–16 operation acceptance creates one event command `{type:"plannerBatch",operations}`, summary `Applied N planner operations`, target `Multiple planner targets`, and ordered flattened changes formatted exactly as `${operationIndex + 1}. ${change}`. It commits one receipt, event/change set, before-state undo basis, and version increment. A well-formed request that reaches an accepted, version-conflict, or domain-rejected decision always reserves one immutable receipt; a rejection commits no workspace, event, version, or undo effect, while schema-invalid input reserves nothing. Apply returns the complete authoritative workspace; domain rejection identifies the zero-based failing `operationIndex`. The current browser request envelope remains singular and keeps its client-generated idempotency UUID. Embedded and global adapters also generate stable request IDs, but request identity is never a model-facing tool argument and actor/provenance is never accepted from browser, model, or UDS JSON. Phase 2 reserves receipt kinds `embedded_codex_apply_planner_operations_v1` and `global_codex_apply_planner_batch_v1`; the latter remains the exact global identity fixed in row 11.

The first forward migration is `002-planner-operations-and-provenance.sql`. Before modifying any existing file database, startup creates an online SQLite backup beside the authority database and verifies that the backup opens and passes `quick_check`; an unverifiable backup aborts only startup before migration. Migration 002 advances both the migration record and the public workspace/export `schemaVersion` from 1 to 2, preserves all existing state/events/receipts/undo bases, backfills structured provenance, and expands the receipt-kind constraint for the two reserved downstream callers. Migration 001 remains immutable.

**Evidence.** `docs/family-readiness-remediation-plan.md` owns the prerequisite single-command authority, while `lib/planner-operation-contract.ts`, `lib/planner-tool-contract.ts`, `server/application/planner-service.ts`, and `tests/planner-tool-contract.test.mjs` implement and prove the tracked follow-up batch/tool contract. The delivery and proof obligations are durable in `docs/codex-agent-runtime-follow-up-phase.md`.

**Historical closure.** Tool names, arguments, limits, results, command visibility, authority, and dependency direction now have one minimal implementation target. The current family release remains single-command; only the follow-up generalizes the shared core so the same `apply` handles one or an atomic ordered batch.

### 7. Durable call/effect lifecycle

**Historical answer.** Extend `chat_turns` with `mode` (`normal|recovery`), `completion_token_hash`, `accepted_effect_count`, `last_effect_sequence`, `recovery_of_turn_id`, and terminal outcome `completed_no_effect|completed_with_effects|failed_no_effect|failed_after_effect|interrupted_no_effect|interrupted_after_effect|recovery_completed|recovery_failed`. Add one `planner_tool_calls` ledger keyed by `(turn_id, tool_call_id)` with sequence, token hash, tool/argument hash, status (`running|succeeded|rejected|cancelled|timed_out|abandoned`), result/effect codes, operation/request/event identities, base/result planner versions, an immutable bounded sanitized replay envelope, and timestamps. Reserve only against a running turn/current token. Exact duplicates replay; changed name/arguments fail. An accepted apply/batch commits workspace, receipt, event, tool row, effect sequence, and turn effect count together before responding. Separate accepted calls are separate effects; an atomic batch is one effect.

Startup marks pre-commit running calls abandoned and the turn interrupted; post-commit calls replay their succeeded ledger row. Reply failure after effects persists `failed_after_effect` and a canonical visible failure. Retry with zero effects creates a new normal turn; Retry after any effect creates a recovery turn with no web or planner tools and only reconstructs the reply from durable effects/current readback. Late/revoked-token calls and terminal replies cannot mutate state.

**Historical evidence.** `server/chat/lifecycle.ts` and `server/chat/embedded-service.ts` own the superseded implemented lifecycle. `tests/embedded-tool-lifecycle.test.mjs` and `tests/integration/dynamic-chat-cutover.test.mjs` exercise its dependent commits, crash boundaries, restart fencing, immutable replay, and recovery-only retry. The current requirements explicitly retire that app-owned conversation retry lifecycle while retaining planner-effect fencing/readback.

**Historical closure.** Every call identity, accepted effect, crash point, terminal outcome, startup action, and Retry mode maps to one durable row/transition and one transaction boundary.

### 8. Research candidate boundary

**Historical answer.** Research emits strict `additionalProperties:false` JSON no larger than 32 KiB: `{schemaVersion:1, candidateId, source:{kind:"web",identity,url,retrievedAt}, title, yieldText?, steps[1..32]}`. The limit is `Buffer.byteLength(JSON.stringify(candidate), "utf8")` after host materialization and complete validation. Source identity/title are 1–200 characters, URL is HTTP(S) and at most 2,048 characters, yield is at most 80, and candidate ID/retrieval time are host materialized for embedded research. Each step has `inputs[0..12]` with amount 1–80 and ingredient 1–160, instruction 1–1,000, and optional duration 1–86,400 seconds; the complete candidate/command has at most 128 inputs across all steps. This aggregate bound matches canonical `Meal.ingredients` without widening ordinary create/update inputs. Amount and ingredient share one non-empty single-line validator and must already equal their trimmed form. There are no HTML, Markdown, page-body, excerpt, arbitrary metadata, or attachment fields. The broader source-reference enum may later include app-owned non-web kinds, but embedded research always emits `web`.

Of research output, the planner context receives only this candidate JSON; separately it receives the frozen user request and bounded canonical planner/transcript context. No candidate/library authority table is added. The shared command is exactly `{type:"replaceMealRecipeFromSource",weekId,mealId,recipe:{title,yieldText?,source:{kind:"web",identity,url,retrievedAt},steps:[{inputs:[{amount,ingredient}],instruction,timerDurationSeconds?}]}}`; it is self-contained and deliberately omits candidate ID. The URL is the standard serialization of a credential-free, fragment-free HTTP(S) `URL` and `retrievedAt` is integer Unix epoch milliseconds. `SourceRecipe` is an informational primary-page reference, not an attestation. Research instructions require one primary starting page, but the host does not claim to detect semantic multi-page blending or textual origin. Embedded admission parses the URL, requires the supplied string to equal standard `URL` serialization byte-for-byte, freezes the tuple without normalizing non-canonical input into acceptance, materializes retrieval time, requires exact equality for preview and apply, and rejects either after full-candidate loss. The command may change only title, yield, and bounded step fields; it cannot change/omit the tuple or write another meal field, and the host does not claim semantic derivation. Global or household callers may submit the same strict command directly with a validated but unattested observation time; actor/admission provenance remains host-only.

Canonical `Meal` stays flat and gains optional `yieldText` plus optional `sourceRecipe` with the exact source tuple. Only `replaceMealRecipeFromSource` may attach or change `sourceRecipe`; `MealPlanInput` must explicitly exclude it, every other command schema excludes it, and ordinary edits preserve it. Generic `updateMealSnapshot` requires `yieldText:string|null`, with null as the only clear signal; replacement omission clears yield. Replacement writes title/yield, ordered shared-service-generated steps, every accepted input flattened in order exactly as `"<amount> <ingredient>"` without deduplication, and source. It preserves ID/date/slot/status/subtitle/venue/protein/prep/leftover/meal notes. Only planned or moved meals in planned or active weeks are eligible. Existing completed steps, instruction notes, running timers, or prep references reject against canonical pre-batch state; the same batch cannot clear and replace, so cleanup must commit separately before a refreshed apply. Rejection has no planner/event/version effect apart from the ordinary immutable rejection receipt. Before planner execution the durable turn stores only the typed compact reference `{schemaVersion:1,candidateId,title,source,stepCount}`. Full candidate memory is not durable; loss before preview/apply requires new no-effect research. Application logs do not retain the full candidate.

**Evidence.** `lib/sourced-recipe-contract.ts`, `server/runtime/codex-follow-up/research-session.ts`, and `server/chat/embedded-service.ts` implement the context split, bounded candidate, exact source binding, and existing-meal replacement path. `tests/sourced-recipe-contract.test.mjs` and `tests/embedded-research-intake.test.mjs` prove the hostile-field and durable-boundary matrix; `docs/functional-spine.md` owns the meal-local recipe semantics.

**Historical closure.** Cross-context shape, informational source-reference semantics, limits, forbidden artifact/field exclusion, transcript behavior, exact existing-meal replacement authority, protected pre-batch state, deterministic mapping, and the only durable intake path are unconditional.

### 4. Dynamic Codex updates and schema compatibility

### 10. Global-client transport
